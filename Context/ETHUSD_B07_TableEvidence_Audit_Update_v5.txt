-----------------------------------------------------------------------
U-01 CRITICAL
No persisted active strategy parameter set exists in the exported database.
-----------------------------------------------------------------------

Evidence:
- strategy_parameter_sets.csv contains 0 rows.
- parameter_activation_history.csv contains 0 rows.

Why this matters:
The B-07 SRS requires versioned, persisted strategy parameters with lifecycle status
(Draft/Candidate/Active/Retired/Rejected), reproducible selection, and activation
history.

With zero rows:
- there is no persisted ACTIVE parameter set,
- there is no activation audit trail,
- the system cannot prove which parameters were used for the captured run,
- replay/optimizer cannot be reproducible against a persisted baseline,
- any IParameterProvider fallback-to-default path becomes operationally opaque.

What this likely means:
At least one of these is true:
A) the runtime never seeded a default parameter set,
B) the admin activation flow was never exercised,
C) rows were created in another database but not in the exported one,
D) repository writes failed or were not committed,
E) export was taken before any parameter operations were performed.

How it should behave:
At minimum, after first startup there should be exactly one ACTIVE default parameter set
persisted in strategy_parameter_sets, and activation history should show the initial
promotion if your design records it.

Required fix:
1) Add a startup seeding path:
   - if strategy_parameter_sets has zero rows, insert the default parameter set,
     mark it ACTIVE, and record the activation event.
2) Enforce exactly one ACTIVE row with a partial unique index or transactional guard.
3) Make ParameterProvider log clearly whether it loaded from DB or fell back to
   hardcoded defaults.
4) Expose the active parameter-set id/hash/version in:
   - logs
   - /api/admin/parameter-sets/active
   - replay_runs
   - optimizer_runs
   - generated signals
5) Add an integration test:
   - start with empty DB
   - run app startup
   - assert one ACTIVE parameter set exists.

Acceptance criteria:
- strategy_parameter_sets export contains at least one row.
- Exactly one row has status = ACTIVE.
- parameter_activation_history contains the initial activation record.
- Runtime logs print the active parameter_set_id and parameter_hash.

U-02 CRITICAL
Replay and optimizer persistence are not evidenced at runtime.
-----------------------------------------------------------------------

Evidence:
- replay_runs.csv = 0 rows
- optimizer_runs.csv = 0 rows
- optimizer_candidates.csv = 0 rows
- optimizer_candidate_folds.csv = 0 rows

Why this matters:
The B-07 SRS requires replay/optimizer execution to be persisted so that results are:
- inspectable,
- reproducible,
- resumable/checkpointed,
- promotable,
- auditable.

Without rows in these tables, the exported runtime state provides no proof that:
- a replay job can start and persist status,
- checkpoint_time is updated,
- an optimizer run writes candidates incrementally,
- fold metrics are captured,
- a best candidate is selected and linked.

What this likely means:
One of the following is happening:
A) the admin replay/optimizer endpoints were never invoked,
B) the service methods exist but do not call the repositories,
C) rows are only created at completion and no completed job ran,
D) writes are rolled back, suppressed, or written to another DB,
E) export occurred before persistence was flushed.

How it should behave:
Replay and optimizer should persist state immediately at run creation, not only on
successful completion.

Minimum expected lifecycle:
Replay:
- insert replay_runs row at launch with status = Running
- periodically update checkpoint_time, candles_read_count, warnings_json
- finalize with status = Completed/Failed

Optimizer:
- insert optimizer_runs row at launch with status = Running
- insert candidates as they are generated
- insert fold rows as each fold completes
- finalize best_candidate_id and best_score at the end

Required fix:
1) Ensure POST /api/admin/replay-runs inserts a replay_runs row before the actual replay
   loop starts.
2) Ensure POST /api/admin/optimizer-runs inserts optimizer_runs immediately.
3) Persist candidates/folds incrementally during execution.
4) Log created ids for replay and optimizer runs.
5) Add end-to-end integration tests that call these admin endpoints and assert row
   creation in all four tables.

Acceptance criteria:
- Starting a replay creates one replay_runs row immediately.
- Starting an optimizer creates one optimizer_runs row immediately.
- Candidate generation creates optimizer_candidates rows before optimizer completion.
- Fold execution creates optimizer_candidate_folds rows.
- CSV export from a smoke run shows non-zero counts in these tables.

U-04 HIGH
Gap tracking is active, but the exported database shows 167 unresolved gaps and this
likely suppresses later regime/signal processing.
-----------------------------------------------------------------------

Evidence:
- gap_events.csv contains 167 rows.
- All 167 rows have resolved = false.
- Later detections occur during runtime, including:
  - detected_at_utc = 2026-03-26 04:44:06.841129+03
  - expected_time = 2026-03-19 04:44:00+03
- regime_snapshots.csv contains only 3 rows total:
  - 03:15, 03:30, 04:30
- candles_15m.csv includes a 04:45 candle,
  but indicator_snapshots.csv has no 15m indicator snapshot at 04:45,
  and regime_snapshots.csv has no 04:45 regime row.

Interpretation:
This strongly suggests one of the following:
A) gap-aware gating is intentionally blocking higher-timeframe processing when a recent
   unresolved gap is detected,
B) gap detection is firing against historical gaps during current runtime and starving
   regime generation,
C) gaps are never resolved/cleared after remediation, so the system can become
   permanently or intermittently self-blocking.

Why this matters:
A gap gate is useful, but if unresolved historical gaps continue to count as blockers
indefinitely, the live strategy may stop producing regimes and signals even when the
current incoming stream is healthy.

How it should behave:
Gap gating should be deterministic and bounded. For example:
- only unresolved gaps inside the effective lookback window for the decision horizon
  should block processing, OR
- only gaps detected after the last successful replay/backfill repair should block
  live evaluation, OR
- repaired gaps should be explicitly marked resolved.

Required fix:
1) Define a gap-resolution lifecycle:
   - created
   - repaired
   - resolved
   - ignored / acknowledged
2) Make the gate query precise:
   - filter by timeframe
   - filter by resolution status
   - filter by effective lookback window
   - optionally filter by detected_at_utc > last_repair_utc
3) Emit structured logs when a regime/signal calculation is skipped because of gaps,
   including the exact gap ids that caused the skip.
4) Add a health endpoint or dashboard widget:
   - unresolved_gap_count_recent
   - oldest_unresolved_gap
   - last_gap_detected_utc
   - last_gap_repaired_utc

Acceptance criteria:
- After a repair/backfill pass, old historical gaps no longer block current decisions.
- A newly detected recent gap does block processing until repaired or acknowledged.
- Logs and dashboard clearly show when and why the gate engaged.

The exported candle data contains impossible OHLC values in closed candles.
-----------------------------------------------------------------------

Evidence from the supplied CSVs:

1m rows with impossible values:
- 2026-03-26 03:16:00+03
  bid_low 2167.73 > min(bid_open 2165.64, bid_close 2167.73)
  ask_low 2169.48 > min(ask_open 2167.39, ask_close 2169.48)
  mid_low 2168.605 > min(mid_open 2166.515, mid_close 2168.605)
- 2026-03-26 03:27:00+03
  bid_high 2162.91 < max(bid_open 2163.48, bid_close 2162.77)
  ask_high 2164.66 < max(ask_open 2165.23, ask_close 2164.52)
  mid_high 2163.785 < max(mid_open 2164.355, mid_close 2163.645)
- 2026-03-26 03:37:00+03
  bid_high 2166.74 < max(bid_open 2166.91, bid_close 2166.68)
  ask_high 2168.49 < max(ask_open 2168.66, ask_close 2168.43)
  mid_high 2167.615 < max(mid_open 2167.785, mid_close 2167.555)

5m rows with impossible values:
- 2026-03-26 03:25:00+03
- 2026-03-26 03:35:00+03
- 2026-03-26 04:40:00+03

15m rows with impossible values:
- 2026-03-25 02:00:00+03
- 2026-03-26 04:30:00+03

Why this is critical:
These are mathematically invalid candles.
Once stored, they can corrupt:
- EMA
- ATR
- ADX
- VWAP-related logic
- regime classification
- signal generation
- replay results
- optimizer scoring

This is a harder failure than "suboptimal logic" because the stored market data itself
becomes invalid.

Likely root cause patterns:
A) rollover/finalization order is still wrong for some minute boundaries,
B) an update path overwrites high/low with the final tick rather than max/min,
C) aggregate builders are deriving 5m/15m from already-corrupted 1m rows,
D) close/finalize logic is being applied to provisional candles incorrectly.

How it should behave:
For every closed candle and every price stream (bid/ask/mid):
- high >= max(open, close)
- low  <= min(open, close)
- high >= low

Required fix:
1) Add a mandatory validation guard before writing any closed candle:
   validate_ohlc(candle)
2) Reject or quarantine any candle that fails validation.
3) Rebuild 5m and 15m strictly from validated, closed 1m rows only.
4) Add a repair job:
   - identify invalid 1m rows,
   - refetch/reconstruct them if possible,
   - delete and rebuild all dependent 5m/15m rows for affected windows,
   - recompute indicators/regimes/signals for the repaired ranges.
5) Add unit tests and integration tests specifically around boundary rollover minutes.

Acceptance criteria:
- Zero closed candles violate OHLC invariants.
- A repair script can scan and report invalid rows.
- Rebuilt 5m/15m rows also satisfy invariants.
