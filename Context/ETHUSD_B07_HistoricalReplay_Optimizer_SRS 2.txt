ETHUSD SCALPING SYSTEM
B-07 REMEDIATION SRS
Historical Replay + Runtime Optimizer Integration
Document Version: 1.0
Prepared For: Implementation after V2.0 audit fixes
Format Target: Plain Text (.txt)

======================================================================
1. PURPOSE
======================================================================

This document defines the remaining implementation required to complete
item B-07 from the audit trail:

    "Wire historical replay and optimizer into runtime."

The purpose of B-07 is to make the system capable of:
1) replaying historical 1m / 5m / 15m market data through the exact same
   regime, indicator, signal, and outcome logic used in live trading,
2) producing deterministic historical signals, outcomes, and feature
   snapshots for training and validation,
3) running parameter optimization on top of those replayed results,
4) activating a selected parameter set safely in runtime,
5) preserving reproducibility, auditability, and rollback.

This is not a cosmetic enhancement. It is the missing mechanism that
turns the application from a live rule engine into an adaptive trading
system with measurable improvement over time.

======================================================================
2. SCOPE
======================================================================

This SRS covers the following deliverables:

A. Historical Replay Engine
   - Replays historical candles through the same business logic used in
     live mode.
   - Reconstructs regime snapshots, indicator snapshots, signals,
     signal_features, and signal_outcomes.

B. Optimizer
   - Evaluates candidate strategy parameter sets against replay results.
   - Uses walk-forward validation rather than naive in-sample fitting.
   - Produces ranked parameter sets with metrics and artifacts.

C. Runtime Integration
   - Adds job orchestration, configuration, persistence, versioning, and
     controlled activation of optimized parameter sets.
   - Enables scheduled and manual optimization runs.

D. Observability + Safety Controls
   - Makes every replay and optimization run reproducible and traceable.
   - Prevents unsafe auto-deploy of overfit or degraded parameter sets.

This SRS does NOT require:
- machine-learning prediction models,
- broker-side auto-order execution,
- multi-symbol portfolio optimization,
- reinforcement learning.

Those may be added later, but they are out of scope for B-07.

======================================================================
3. PROBLEM STATEMENT
======================================================================

The current system generates live signals using a fixed strategy version
and fixed thresholds. Even if all V2.0 bug fixes were applied, the
system still lacks a closed feedback loop.

Without B-07:
- historical candles exist but are not replayed through runtime logic,
- strategy performance cannot be recomputed deterministically,
- feature importance cannot be updated using actual outcomes,
- thresholds and weights remain static,
- there is no formal mechanism to promote improved parameter sets.

Therefore, B-07 must establish an offline-to-online feedback pipeline.

======================================================================
4. HIGH-LEVEL GOALS
======================================================================

G1. Replay historical data through the SAME domain services used by live
    processing, not through an alternate simplified code path.

G2. Ensure deterministic results:
    same historical input + same parameter set + same code version
    => same signals, same outcomes, same metrics.

G3. Persist all artifacts needed for audit and later analysis:
    parameter sets, run definitions, metrics, folds, candidate rankings.

G4. Use walk-forward validation so the optimizer does not merely fit the
    past.

G5. Allow manual approval OR rule-based guarded activation of a new
    parameter set.

G6. Make the system safe:
    no active parameter set may be replaced unless promotion criteria are
    satisfied.

======================================================================
5. ARCHITECTURAL PRINCIPLES
======================================================================

P1. Single Source of Trading Logic
    RegimeAnalyzer, IndicatorService, SignalEngine, RiskManager, and
    OutcomeEvaluator must be reused by replay. Do NOT duplicate rules.

P2. Separation of Concerns
    Replay, optimization, runtime activation, and reporting must be
    separate modules communicating through explicit contracts.

P3. Idempotence
    The same replay window for the same symbol, strategy version, and
    parameter set may be rerun safely without corrupting state.

P4. Reproducibility
    Every replay/optimizer result must be tied to:
      - code version / git hash if available,
      - strategy version,
      - parameter set id,
      - data window,
      - data checksum or row counts,
      - objective function version.

P5. Promotion Safety
    A candidate must not auto-replace the current parameter set if:
      - out-of-sample performance is worse than baseline,
      - sample size is insufficient,
      - drawdown exceeds cap,
      - parameter drift exceeds allowed bounds,
      - recent live degradation is unresolved.

======================================================================
6. TARGET OPERATING MODEL
======================================================================

The finished system must support three modes:

6.1 Live Mode
    Existing runtime processing generates current signals using the
    active strategy parameter set.

6.2 Replay Mode
    Historical data in a specified date range is replayed through the
    exact same core logic, generating historical artifacts.

6.3 Optimization Mode
    Candidate parameter sets are evaluated by repeated replay across
    train/validation folds; best candidates are ranked and optionally
    promoted.

7.4 Historical Replay Engine
----------------------------------------------------------------------

REQ-B07-004
Implement a HistoricalReplayService that replays historical data in
time order.

Required inputs:
- symbol
- start_utc
- end_utc
- parameter_set_id
- replay mode
- overwrite policy

Required outputs:
- regime_snapshots
- indicator_snapshots
- signals
- signal_features
- signal_outcomes (where finalizable)
- replay_runs progress and summary

Replay algorithm requirements:
1) Load closed 1m candles ordered by open_time ascending.
2) Aggregate or read 5m and 15m data consistently from closed 1m candles.
3) Advance a simulated clock candle-by-candle.
4) On each 1m close:
   - update internal bar state,
   - detect completed 5m / 15m closes,
   - run regime first on 15m close,
   - run signal logic on 5m close,
   - use only data available up to that point.
5) When a signal is emitted:
   - persist signal,
   - persist full feature vector,
   - record provisional outcome status if future bars are insufficient.
6) Finalize outcomes only when enough future candles exist.
7) Never use future data in decision computation.

Critical rule:
Replay must be "causally correct". At bar t, the engine may only inspect
bars <= t.

7.6 Outcome Reconstruction
----------------------------------------------------------------------

REQ-B07-006
Historical replay must reconstruct signal outcomes using the same
OutcomeEvaluator rules used live.

For each signal s generated at time t0:
- entry price = signal.entry_price
- SL = signal.sl
- TP = signal.tp
- future bars = all closed 5m bars after t0 up to timeout horizon

The engine must determine:
- whether TP was hit first,
- whether SL was hit first,
- whether timeout was reached,
- MFE (maximum favorable excursion),
- MAE (maximum adverse excursion),
- pnl_r,
- status = WIN / LOSS / EXPIRED / PENDING

Definitions:

For BUY:
  favorable_move_i = High_i - Entry
  adverse_move_i   = Entry - Low_i

For SELL:
  favorable_move_i = Entry - Low_i
  adverse_move_i   = High_i - Entry

Then:
  MFE = max(favorable_move_i)
  MAE = max(adverse_move_i)

R multiple:
  risk_per_unit = abs(Entry - SL)
  pnl_r =
    +TargetRMultiple if TP hit first
    -1.0             if SL hit first
    ((Close_timeout - Entry) / risk_per_unit) for timed exit BUY
    ((Entry - Close_timeout) / risk_per_unit) for timed exit SELL

If futureCandles.Count < TimeoutBars:
  outcome MUST remain PENDING

This requirement must exactly align with the V2.0 audit fix for B-02.

7.10 Walk-Forward Validation
----------------------------------------------------------------------

REQ-B07-010
The optimizer MUST NOT evaluate candidates on a single in-sample period
only. It must use walk-forward validation.

Required minimum design:
- Split the historical window into K ordered folds.
- For each fold k:
  train on window T_k
  validate on immediate future window V_k
- Aggregate out-of-sample metrics across validation folds.

Example:
- 180 days total data
- Fold 1: train days 1-90, validate days 91-120
- Fold 2: train days 31-120, validate days 121-150
- Fold 3: train days 61-150, validate days 151-180

Candidate score must be computed from validation metrics, not train
metrics.

----------------------------------------------------------------------
7.11 Objective Function
----------------------------------------------------------------------

REQ-B07-011
The optimizer must compute an explicit objective score for each candidate.

Minimum metrics to compute:
- trade_count
- win_rate
- avg_pnl_r
- total_pnl_r
- profit_factor
- max_drawdown_r
- expectancy_r
- sharpe_like_r (optional initial version)
- exposure_days
- signal_density
- timeout_rate
- no_trade_rate

Definitions:

win_rate = wins / max(1, completed_trades)

expectancy_r = (win_rate * avg_win_r) - ((1 - win_rate) * avg_loss_r_abs)

profit_factor = gross_win_r / max(epsilon, gross_loss_r_abs)

max_drawdown_r:
  Let E_t = cumulative pnl_r after trade t
  peak_t = max(E_1 ... E_t)
  drawdown_t = peak_t - E_t
  max_drawdown_r = max(drawdown_t)

Suggested objective score:
  Score(theta) =
      w1 * Normalize(total_pnl_r)
    + w2 * Normalize(expectancy_r)
    + w3 * Normalize(profit_factor)
    + w4 * Normalize(win_rate)
    - w5 * Normalize(max_drawdown_r)
    - w6 * Normalize(timeout_rate)
    - w7 * OverfitPenalty(theta)
    - w8 * SparsityPenalty(theta)

Where:
- Normalize(x) rescales metric x to comparable range across candidates.
- OverfitPenalty(theta) = max(0, train_score - val_score)
- SparsityPenalty(theta) applies when trade_count is too low.

Recommended initial hard filters before scoring:
- validation completed trades >= 30
- profit_factor >= 1.05
- expectancy_r > 0
- max_drawdown_r <= configured limit
- signal density within reasonable band

If a candidate fails hard filters, mark as REJECTED.

----------------------------------------------------------------------
10.1 HistoricalReplayService
----------------------------------------------------------------------

Responsibilities:
- load candles,
- feed candle stream into domain pipeline,
- persist generated artifacts,
- checkpoint progress,
- finalize outcomes.

Pseudo-flow:

for each candle_1m in closedCandlesOrdered:
    replayClock = candle_1m.CloseTime
    update 1m state
    if completes 5m bar:
        build closed 5m
        compute indicators on closed bars only
    if completes 15m bar:
        compute regime first
        persist regime snapshot
    if completes 5m bar:
        evaluate signal using current 15m regime
        if signal emitted:
            persist signal + features
            try finalizing if enough future bars available
    checkpoint every N bars

Key requirement:
Replay must simulate the same event ordering as live fixed by B-03:
15m regime update before 5m signal generation on shared boundaries.

11. MATHEMATICAL AND STATISTICAL DETAILS
======================================================================

11.1 Expected Value / Expectancy
  expectancy_r = mean(pnl_r_i)

Alternative decomposed form:
  expectancy_r = p(win) * avg_win_r - p(loss) * avg_loss_r_abs

11.2 Profit Factor
  PF = sum(all positive pnl_r) / max(epsilon, abs(sum(all negative pnl_r)))

11.3 Maximum Drawdown
  Let cumulative equity:
      E_0 = 0
      E_t = E_(t-1) + pnl_r_t

  Running peak:
      P_t = max(E_0 ... E_t)

  Drawdown:
      DD_t = P_t - E_t

  Max drawdown:
      MDD = max(DD_t)

11.4 Overfit Penalty
  OP = max(0, TrainScore - ValidationScore)

11.5 Sparsity Penalty
  Let n = completed validation trades
  Let n_min = required minimum trades

  SP = max(0, (n_min - n) / n_min)

11.6 Composite Objective Example
  Score =
      0.30 * Norm(total_pnl_r)
    + 0.20 * Norm(expectancy_r)
    + 0.20 * Norm(profit_factor)
    + 0.10 * Norm(win_rate)
    - 0.10 * Norm(max_drawdown_r)
    - 0.05 * timeout_rate
    - 0.03 * overfit_penalty
    - 0.02 * sparsity_penalty

These weights are initial defaults and must themselves be versioned.

17. RISKS AND MITIGATIONS
======================================================================

Risk R1: Overfitting
Mitigation:
- walk-forward validation
- baseline comparison
- overfit penalty
- manual approval for first release

Risk R2: Runtime instability after activation
Mitigation:
- last-known-good fallback
- atomic cache refresh
- activation history + quick rollback

Risk R3: Historical data quality issues distort optimization
Mitigation:
- unresolved gap gating
- data-quality warnings
- replay artifact counts
- sample sufficiency thresholds

Risk R4: Performance bottlenecks
Mitigation:
- checkpointing
- batched persistence
- per-fold summaries
- candidate count limits

Risk R5: Hidden logic divergence between live and replay
Mitigation:
- shared domain services
- golden dataset regression
- code review rule: no duplicate rule engines

18. DEFINITION OF DONE FOR B-07
======================================================================

B-07 is complete only when ALL of the following are true:

1) Live trading parameters are externally versioned and runtime-resolved.
2) Historical replay can rebuild signals/features/outcomes deterministically.
3) Optimizer can evaluate baseline and candidates using walk-forward folds.
4) Candidate metrics and artifacts are persisted and auditable.
5) A new parameter set can be promoted manually without restart.
6) Safety gates prevent degraded/overfit auto-promotion.
7) Full end-to-end tests pass on a golden dataset.
8) Admin APIs and run summaries exist for operational control.

If any of the above is missing, B-07 is not complete.

20. FINAL IMPLEMENTATION GUIDANCE FOR CODE-GENERATING AGENTS
======================================================================

The implementing agent must follow these rules:

- Reuse existing domain services wherever possible.
- Do not create a second "approximate" signal engine for replay.
- Do not activate candidates automatically in first release.
- Prefer deterministic, testable components over clever abstractions.
- Every phase must include unit + integration tests before next phase.
- Every persisted artifact must include parameter_set_id and strategy_version.
- Every candidate evaluation must use out-of-sample metrics.
- Never let optimization logic use future data beyond each validation window.
