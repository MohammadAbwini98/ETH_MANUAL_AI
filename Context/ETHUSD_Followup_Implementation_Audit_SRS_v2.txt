1. PURPOSE
====================================================================

This document is a follow-up corrective Software Requirements Specification
(SRS) and implementation audit for the second uploaded ETH.zip package,
including source code, logs, and database CSV exports in the
db_export_20260326_110344 folder.

This audit specifically checks:
1. whether the issues from the previous audit were fixed,
2. whether new defects or regressions were introduced,
3. whether the implementation now matches the intended phased design,
4. what still blocks correctness, robustness, and SRS compliance,
5. what exact changes are still required before trusting the system.

This document is written to be used as a repair specification for an
engineer or an AI coding agent.

Important limitation:
- This review is based on static source inspection, log review, and CSV
  evidence.
- The system was not executed locally because the .NET runtime is not
  available in this environment.
- Conclusions are therefore grounded in code logic plus runtime artifacts,
  not a fresh replay.

====================================================================
3. EXECUTIVE SUMMARY
====================================================================

Overall assessment:
- This version is materially better than the prior one.
- Several critical issues from the previous audit HAVE been fixed.
- Signals are now generated, stored, feature rows are inserted, and
  outcomes are persisted.
- The live engine now derives 5m and 15m candles from closed 1m candles
  instead of constructing all timeframes independently from tick polling.

However:
- the implementation is STILL NOT fully aligned with the intended SRS,
- several important correctness defects remain,
- one of the most serious remaining defects is that signal outcomes are
  being marked EXPIRED too early,
- another serious defect is that the generic candle upsert repository
  still auto-closes candles based on list position, which corrupts
  partial higher-timeframe handling during backfill,
- and the system still does not deliver true tick-by-tick market updates.

Bottom line:
- This build is now good enough to demonstrate end-to-end flow.
- It is NOT yet good enough to trust for production trading decisions or
  to use as a clean training dataset for "smart" optimization.

ISSUE B-01
Severity: CRITICAL
Title: Generic bulk upsert still auto-closes candles based on list order,
       which corrupts partial higher-timeframe handling
--------------------------------------------------------------------

Code locations:
- src/EthSignal.Infrastructure/Db/CandleRepository.cs lines 29-30 and 57-60
- src/EthSignal.Infrastructure/Engine/BackfillService.cs lines 100-132
- tests/EthSignal.Tests/Infrastructure/CandleRepositoryTests.cs lines 116-125

Current implementation:
- BulkUpsertAsync sets is_closed using:
    c.OpenTime < lastTime || c.IsClosed
- This means every candle in a multi-row batch except the last row is
  automatically written as closed, even if c.IsClosed is false.

Why this is wrong:
- A generic repository method must not silently override closure state
  based on row ordering.
- Closure state must come from the candle object itself.
- BackfillService explicitly marks incomplete higher-timeframe candles as
  IsClosed = false, but BulkUpsertAsync can still flip them to true when
  they are not the final row in the batch.

Data evidence:
- logs show:
  - Stored 5m: 1985 rows (1983 complete, 2 partial)
  - Indicators 5m: 1985 snapshots computed
- If 2 of the 5m rows were really partial, indicators should have been
  computed only on the 1983 complete rows.
- The same issue appears in 15m:
  - Stored 15m: 664 rows (655 complete, 9 partial)
  - Indicators 15m: 664 snapshots computed
- Therefore the "partial" rows were effectively treated as closed before
  indicator generation.

Impact:
- Incomplete higher-timeframe candles can enter indicator history.
- Regime history can be computed on incomplete 15m data.
- Signal and backtest results can become contaminated.
- The repository test currently codifies this incorrect behavior by
  expecting the last row semantics instead of explicit IsClosed semantics.

Required correction:
1. In CandleRepository.BulkUpsertAsync, replace:
      c.OpenTime < lastTime || c.IsClosed
   with:
      c.IsClosed
2. Remove the hidden "all except last are closed" behavior entirely.
3. Update BackfillService to set IsClosed explicitly for each candle.
4. Fix CandleRepositoryTests so they no longer assume automatic closure
   for every row except the last.

How it should behave:
- Repository methods must be passive storage layers.
- The caller decides which candles are closed or open.
- Partial 5m/15m candles produced by backfill must stay is_closed=false
  until completed later from correct data.

Re-test requirements:
- Unit test: a batch containing [closed, open, closed] must preserve all
  three flags exactly.
- Integration test: backfill a range that contains partial 5m/15m buckets;
  verify those rows stay is_closed=false.
- Indicator test: GetClosedCandlesAsync must exclude those partial rows.

ISSUE B-02
Severity: CRITICAL
Title: Outcome evaluator marks signals EXPIRED too early instead of PENDING
--------------------------------------------------------------------

Code locations:
- src/EthSignal.Infrastructure/Engine/OutcomeEvaluator.cs lines 31-131
- src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 412-420
- tests/EthSignal.Tests/Engine/OutcomeEvaluatorTests.cs lines 107-121

Current implementation:
- OutcomeEvaluator.Evaluate loops through the available future candles.
- If TP/SL is not hit, it returns EXPIRED at the end even if the number of
  observed candles is far below TimeoutBars.

Why this is wrong:
- A signal should remain PENDING until one of these is true:
  1. TP hit,
  2. SL hit,
  3. TimeoutBars candles were fully observed.
- The current logic expires trades prematurely.

Runtime evidence:
- logs show:
  - 2026-03-26 09:30 signal generated
  - 2026-03-26 09:45 signal resolved: EXPIRED PnL=0.68R
- That is only 15 minutes later, i.e. 2 five-minute bars observed.
- TimeoutBars is 60, so expiration after 2 bars is incorrect.

Data evidence:
- signal_outcomes.csv for signal 758c33d6-...:
  - bars_observed = 2
  - outcome_label = EXPIRED
- That is logically impossible under the intended timeout model.

Impact:
- Signals close far too early.
- Session risk limits are distorted.
- Win rate, average R, and optimization data are wrong.
- Follow-up signals may become allowed too soon or too late depending on
  false expirations.

Required correction:
1. In OutcomeEvaluator.Evaluate:
   - if TP/SL not hit and futureCandles.Count < TimeoutBars, return
     OutcomeLabel.PENDING, not EXPIRED.
2. Only return EXPIRED when:
   - futureCandles.Count >= TimeoutBars and TP/SL still not hit.
3. In LiveTickProcessor.EvaluateOpenSignalsAsync:
   - only persist/update outcomes when label != PENDING.
4. Update tests:
   - the current Timeout_Returns_EXPIRED test with only 3 candles is wrong
     and should be replaced with two tests:
       a. fewer than TimeoutBars without TP/SL => PENDING
       b. exactly TimeoutBars without TP/SL => EXPIRED

How it should behave:
- Open signals stay OPEN until fully resolved or timed out.
- EXPIRED is a terminal state only after the configured timeout window.

Re-test requirements:
- Create a signal with 3 future candles, no TP/SL hit -> must remain PENDING.
- Create the same signal with 60 future candles, no TP/SL hit -> must be EXPIRED.
- Verify session blocking logic uses the correct set of still-open signals.

ISSUE B-03
Severity: HIGH
Title: Signal generation on 15m boundaries still uses stale regime
--------------------------------------------------------------------

Code locations:
- src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 147-177
- src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 231-254

Current implementation:
- On every completed 1m candle, the code checks M5 first and M15 second:
    foreach (var tf in new[] { Timeframe.M5, Timeframe.M15 })
- When both boundaries close at the same timestamp (for example 09:30, 10:00,
  10:15, etc.), the 5m candle is processed first.
- That means TryGenerateSignal can run before the newly closed 15m candle has
  updated the regime.

Runtime evidence:
- log order at 09:30:
  - 5m candle closed
  - indicator computed: 5m
  - Signal generated: SELL
  - 15m candle closed
  - indicator computed: 15m
  - Regime updated: BEARISH
- The signal was therefore generated using the PREVIOUS regime, not the regime
  derived from the latest closed 15m bar.

Why this is wrong:
- At a shared 5m/15m boundary, the freshest 15m close is already available.
- The current 5m signal should use the newest 15m regime, not the previous one.

Impact:
- The first 5m signal after a 15m close can be generated with stale regime
  direction, stale score, or stale disqualifiers.

Required correction:
Option A (preferred):
1. When both M5 and M15 close on the same event, process M15 first,
   then process M5 signal generation.

Option B:
1. Detect shared boundary.
2. Compute/update regime before calling TryGenerateSignal for that 5m close.

How it should behave:
- For any 5m candle that coincides with a 15m close, the regime filter used
  by SignalEngine must be based on the just-closed 15m candle.

Re-test requirements:
- Synthetic test at a 15m boundary where prior regime differs from new regime.
- Verify the resulting 5m signal uses the new regime, not the old one.

ISSUE B-04
Severity: HIGH
Title: Current indicators API still violates the closed-candle policy
--------------------------------------------------------------------

Code location:
- src/EthSignal.Web/Program.cs lines 140-148

Current implementation:
- /api/indicators/current loads closed candles, then appends the current open
  candle using:
    open with { IsClosed = true }
- It then computes "current" indicators on that manipulated series.

Why this is wrong:
- The SRS and the prior audit required indicator calculations to be based on
  closed candles only unless explicitly labeled provisional.
- Forcing an open candle to IsClosed=true hides the fact that it is provisional.
- This reintroduces a narrower version of the earlier "open treated as closed"
  defect.

Additional practical problem:
- The live engine only persists open 1m candles.
- Open 5m and 15m candles are not persisted continuously.
- Therefore this endpoint is not actually giving true live 5m/15m indicators;
  it often falls back to the latest stored closed snapshot anyway.

Impact:
- Dashboard values can be unstable or misleading.
- Users cannot tell whether a displayed indicator is final or provisional.

Required correction:
Choose one of these designs and stick to it:

Design A (strict closed-candle policy):
- /api/indicators/current returns only the latest NON-provisional stored
  5m and 15m snapshots.

Design B (explicit live/provisional mode):
- Compute a provisional snapshot from the current open higher-timeframe candle,
  but return:
  - isProvisional = true
  - live snapshot fields clearly marked as provisional
- Do NOT fake IsClosed=true.

How it should behave:
- Closed and provisional indicator modes must be explicit and never mixed.

Re-test requirements:
- API response must include isProvisional for live computations.
- UI must visually distinguish closed vs provisional values.

ISSUE B-05
Severity: HIGH
Title: Gap detection exists but gap-aware trading controls are still missing
--------------------------------------------------------------------

Code locations:
- src/EthSignal.Infrastructure/Engine/BackfillService.cs lines 173-181
- signal/regime/indicator code paths contain no gap gate afterward

Data evidence:
- logs show: Detected 157 gaps in 1m candles.
- gap_events.csv contains many rows.

Current implementation:
- The system records gaps but still proceeds to compute indicators, classify
  regime, and generate signals across the dataset.

Why this is wrong:
- 157 one-minute gaps in a seven-day crypto dataset are significant.
- Indicators such as EMA, RSI, ATR, ADX, and VWAP assume contiguous time
  series. If the series has missing bars and the engine does not reset or
  block affected windows, the outputs are mathematically inconsistent with
  the intended formulas.

Impact:
- Regime history can be computed across broken data.
- Signals and outcomes can be labeled on unreliable indicator states.
- Any learning pipeline trained on these rows will inherit bad labels.

Required correction:
1. Add a "data quality gate" before indicator/regime/signal evaluation.
2. For each new 5m and 15m candle, determine whether the underlying 1m window
   contains gaps.
3. If a recent gap exists inside the lookback window:
   - either block signal generation,
   - or reset/re-warm indicators from the first continuous segment after the gap.
4. Expose a gap-health flag in the dashboard/API.

How it should behave:
- The engine must never treat a broken time series as continuous.

Re-test requirements:
- Inject a missing 1m bar into a test window.
- Verify regime and signal generation are either blocked or explicitly marked
  invalid until warm-up continuity is restored.

Re-test requirements:

Code locations:

Current implementation:

- Historical replay should accelerate dataset growth without contaminating

Re-test requirements:

- Verify the stored replay signals match deterministic expectations.
- Verify optimizer output is reproducible on the same dataset.

ISSUE B-08
Severity: MEDIUM
Title: Daily drawdown logic uses net PnL, not realized losses only
--------------------------------------------------------------------

Code location:
- src/EthSignal.Infrastructure/Engine/RiskManager.cs lines 93-115

Current implementation:
- The comment says:
    "Daily drawdown: sum of realized losses today"
- But the code actually sums ALL outcome PnL:
    dailyPnlR += o.PnlR
- This means earlier wins reduce the measured drawdown.

Why this is wrong:
- For a protective trading stop, daily drawdown should be based on:
  - realized losses only, or
  - true equity peak-to-trough drawdown.
- Netting wins against losses can allow too much additional risk after a
  bad sequence.

Impact:
- Session risk protection is weaker than intended.

Required correction:
Choose one consistent policy:
Policy A (simple and conservative):
- dailyLossR = sum(abs(PnlR)) for all negative outcomes only
- dailyDrawdownPct = dailyLossR * RiskPercentPerTrade

Policy B (more realistic):
- reconstruct intraday equity curve from ordered outcomes
- compute max drawdown from peak equity

How it should behave:
- A profitable trade earlier in the day should not fully erase the risk
  control intended to stop further losses.

Re-test requirements:
- Build a test with WIN, LOSS, LOSS sequence and verify drawdown policy
  matches the chosen risk design.

ISSUE B-11
Severity: MEDIUM
Title: Market structure logic is still an oversimplified proxy
--------------------------------------------------------------------

Code location:
- src/EthSignal.Infrastructure/Engine/RegimeAnalyzer.cs lines 154-196

Current implementation:
- Structure is inferred from recent CLOSE prices, not swing highs/lows.
- For bullish structure it returns true if HH OR HL.
- For bearish structure it returns true if LH OR LL.

Why this is weak:
- The intended concept was HH/HL for bull and LH/LL for bear.
- Using closes instead of actual swing highs/lows is a simplification.
- Using OR instead of AND makes the test much looser than the stated regime
  definition.

Impact:
- Regime structure confirmation can become noisy or too permissive.

Required correction:
1. Base market structure on actual candle highs/lows, not snapshot closes.
2. For bullish structure require both:
   - higher high
   - higher low
3. For bearish structure require both:
   - lower high
   - lower low
4. If a fallback trend approximation is desired, expose it as a separate
   weaker condition instead of mixing it with true structure.

Re-test requirements:
- Add tests for:
  - HH only, no HL -> should fail strict bullish structure
  - HL only, no HH -> should fail strict bullish structure
  - same for bearish cases
