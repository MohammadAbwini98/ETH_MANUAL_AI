1. PURPOSE
====================================================================

This document is a corrective Software Requirements Specification (SRS)
and implementation audit for the uploaded ETH.zip project, the attached
logs, and the exported CSV tables.

Its purpose is to:
1. compare the current implementation against the intended phased SRS,
2. identify mismatches, logical errors, wrong mappings, missing pieces,
3. explain why no signals were generated yet,
4. define the exact corrective actions and re-test requirements needed
   before the system can be considered reliable.

This document is written to be used later by an engineer or an AI coding
agent to fix the implementation systematically.

Important limitation:
- This audit is based on static source review, uploaded logs, and CSV
  exports.
- The code was not executed end-to-end in this environment because the
  .NET SDK/runtime was not available here.
- Findings below are therefore based on code logic, data evidence, and
  logs, not on a local runtime replay.

3. EXECUTIVE SUMMARY
====================================================================

Overall assessment:
- The project contains most of the major modules required by the target
  architecture:
  - Capital.com API client
  - PostgreSQL repositories
  - backfill engine
  - live tick processor
  - indicator engine
  - regime analyzer
  - signal engine
  - risk manager
  - outcome evaluator
  - dashboard pages and APIs
  - unit/integration test scaffolding
- However, the implementation is NOT yet aligned with the intended SRS
  strongly enough to trust it for live trading recommendations.

Current status by phase:
- Phase 1 (ingestion correctness): PARTIALLY IMPLEMENTED but NOT PASSING
- Phase 2 (indicator engine): PARTIALLY IMPLEMENTED but BLOCKED by data quality issues
- Phase 3 (regime classification): IMPLEMENTED but too narrow and not historically persisted
- Phase 4 (signal generation): IMPLEMENTED but not aligned to the intended entry model and over-gated
- Phase 5 (risk management): PARTIALLY IMPLEMENTED; core stop/TP math exists but daily risk policy is missing
- Phase 6 (signal outcomes and learning): PARTIALLY IMPLEMENTED; outcomes depend on signals, and feature storage is missing
- Phase 7 (optimization / adaptive learning): LIBRARY EXISTS but not wired into runtime or persistence
- Phase 8 (dashboard): IMPLEMENTED visually, but the live data model is not tick-by-tick and adds API load

Top 7 reasons the system is not ready:
1. 5m and 15m candles are not consistently derived from 1m truth.
2. The live rollover logic still closes old candles using the first price
   of the next candle.
3. Open candles are read back from the repository as if they were closed.
4. Backfill creates partial higher-timeframe candles and force-closes them.
5. The dashboard/API layer directly polls Capital.com in parallel with the
   ingestion loop, increasing rate-limit risk and creating data divergence.
6. The signal engine only emits live signals on fresh 5m closes, while the
   observed regime remained NEUTRAL during the captured runtime.
7. The learning loop is incomplete because signal_features are never stored.

Bottom line:
- The absence of signals is currently expected from the captured runtime,
  but it also hides deeper architectural issues that would still reduce
  signal quality even after the market becomes directional.

====================================================================
4. WHY THERE ARE NO SIGNALS YET
====================================================================

There is not one single reason; there are multiple stacked reasons.

4.1 Immediate runtime reason observed in logs and data
------------------------------------------------------

During the captured run:
- the 15m regime remained NEUTRAL,
- the regime score remained low,
- ADX remained below the regime threshold,
- therefore the signal engine returned NO_TRADE before risk or persistence.

Evidence from logs:
- regime loaded as NEUTRAL at startup,
- regime updated again as NEUTRAL after 15m closes,
- no "Signal generated" entries appear in the uploaded log.

Evidence from data:
- regime_snapshots.csv contains only NEUTRAL rows,
- recent 15m disqualifying conditions repeatedly include ADX < 18,
- recent 5m ADX values are around 10 to 12, which is too weak for the
  current regime and signal thresholds.

4.2 Structural reason
---------------------

Signals are only generated inside the live loop when a 5m candle closes:
- src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 201-258

Backfill computes indicators and latest regime, but it does NOT generate
historical signals into the signals table.
Therefore:
- if the app did not stay online long enough during a directional regime,
  signal tables remain empty,
- there is no historical bootstrap of signals from already stored candles.

4.3 Strategy-gating reason
--------------------------

Even if the regime became directional, the current signal engine is strict:

Mandatory gates:
- regime must not be NEUTRAL,
- EMA condition must pass,
- RSI condition must pass,
- MACD condition must pass,
- spread condition must pass,
- at least one confirmation must pass,
- final score must be at least 70.

Because the current design also requires a trending regime and uses an ADX
threshold of 18, signals will be rare in choppy conditions.

4.4 Data-quality reason
-----------------------

Because higher-timeframe candles and live volume are not consistently
derived from finalized 1m candles, even a valid trade setup can fail the
signal rules incorrectly. This can suppress signal creation.

4.5 Missing learning data reason
--------------------------------

signal_features, signal_outcomes, and signals are empty because:
- no live signal passed all gates yet,
- feature rows are never inserted anywhere in runtime code,
- the optimization engine exists only as an offline utility.

Conclusion:
- The immediate reason for no signals is NEUTRAL regime and weak ADX.
- The deeper reasons are architecture and data correctness gaps that must
  be fixed before trusting future signals.

PHASE 1 - DATA INGESTION, CANDLE CORRECTNESS, AND PERSISTENCE
--------------------------------------------------------------------

ISSUE P1-01
Severity: CRITICAL
Title: Higher-timeframe live candles do not reliably equal aggregation of 1m candles

Current implementation:
- The live loop updates 1m, 5m, and 15m candles directly from spot ticks in
  parallel.
- 5m and 15m candles are not derived exclusively from finalized 1m candles.
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 105-135

Why this is wrong:
- The SRS required 5m and 15m candles to be internally consistent with the
  1m base series.
- Directly building higher-timeframe candles from spot polling introduces
  divergence in:
  - open price,
  - high/low,
  - close,
  - volume.

Data evidence:
- candles_5m.csv and candles_15m.csv do not fully match aggregation of
  candles_1m.csv.
- Mismatches were found in OHLC and volume, especially in recent live-built
  bars.

Impact:
- indicator values become inconsistent,
- signal decisions can differ from what the 1m truth would imply,
- backtests and live behavior become incomparable.

Required fix:
- Make 1m the only live-constructed candle stream.
- Derive 5m and 15m only by aggregating CLOSED 1m candles.
- Do not update higher-timeframe OHLC directly from ticks.
- Trigger:
  - on 1m close -> persist 1m
  - if 1m close completes a 5m bucket -> aggregate last 5 closed 1m rows
  - if 1m close completes a 15m bucket -> aggregate last 15 closed 1m rows

Re-test requirement:
- SQL and unit test must prove that every 5m candle equals aggregation of
  the exact 5 underlying 1m candles.
- Same for 15m.

ISSUE P1-02
Severity: CRITICAL
Title: Rollover logic closes the old candle with the first price of the new candle

Current implementation:
- On boundary crossing, the code finalizes the current candle by calling:
  - UpdateCandle(current, spot)
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 110-123

Why this is wrong:
- The first tick after the boundary belongs to the NEW candle.
- The previous candle should close using the LAST price seen before the
  boundary, not the first price of the next bucket.
- This is the same defect identified in the earlier review.

Additional test problem:
- The test suite currently validates the wrong behavior.
- Relevant test:
  - tests/EthSignal.Tests/Engine/LiveRolloverTests.cs lines 39-47
- That test explicitly closes the prior candle using the boundary tick price.

Impact:
- previous candle close is polluted,
- higher-timeframe aggregation is distorted,
- indicators and signals are shifted.

Required fix:
- Track previous bid/ask tick.
- On bucket transition:
  - finalize old candle using previous tick,
  - create new candle using current tick.
- Update tests to reflect the corrected rollover model.

Correct target behavior:
- old candle close = last tick strictly before new bucket open
- new candle open = first tick at or after new bucket open

Re-test requirement:
- deterministic unit tests for 1m, 5m, and 15m transitions.
- verify no candle uses a price from the next bucket as its final close.

ISSUE P1-03
Severity: CRITICAL
Title: Repository reads open candles back as if they were closed

Current implementation:
- GetOpenCandleAsync() does not select is_closed.
- ReadCandle() hardcodes IsClosed = true.
- Relevant code:
  - src/EthSignal.Infrastructure/Db/CandleRepository.cs lines 102-121
  - src/EthSignal.Infrastructure/Db/CandleRepository.cs lines 320-335

Why this is wrong:
- Any caller receiving an "open" candle from the repository sees it as
  closed/final.
- This directly breaks closed-candle policy.

Impact:
- open candle indicator snapshots may be treated as final,
- API responses can misrepresent live data,
- provisional/final distinction is compromised.

Required fix:
- Include is_closed in all select statements that read candles.
- Populate RichCandle.IsClosed from the database value.
- Create separate read methods if needed:
  - ReadClosedCandle
  - ReadOpenCandle

Re-test requirement:
- repository tests must verify:
  - open candle returns IsClosed = false,
  - closed candle returns IsClosed = true.

ISSUE P1-04
Severity: CRITICAL
Title: Backfill creates partial 5m and 15m candles and then force-closes them

Current implementation:
- Backfill fetches only the requested 1m range,
- aggregates whatever 1m rows were fetched into 5m/15m,
- then force-closes all open candles.
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/BackfillService.cs lines 98-108
  - src/EthSignal.Infrastructure/Db/CandleRepository.cs lines 233-245

Why this is wrong:
- If backfill starts or ends inside a higher-timeframe bucket, the first or
  last 5m/15m candle can be partial.
- Example observed in data:
  - first 5m candle at 2026-03-18 02:05 was built from only the minutes
    02:07-02:09 because earlier minutes were not fetched.
- Yet these candles are stored as closed/final.

Impact:
- higher-timeframe OHLC and volume become invalid,
- indicators on early/late edges of a backfill window become wrong,
- regime and signal logic start from corrupted data.

Required fix:
- When backfilling 5m/15m derived candles, always pull a boundary-safe
  overlap:
  - start from floor(fromUtc, 15m) minus 15m,
  - end at floor(toUtc, 1m) but do not finalize incomplete current buckets.
- Alternatively:
  - persist only 1m from backfill,
  - rebuild 5m/15m from the full 1m table with strict completeness checks.

Completeness rule:
- mark a 5m candle closed only if 5 distinct 1m closed rows exist,
- mark a 15m candle closed only if 15 distinct 1m closed rows exist.

Re-test requirement:
- first and last aggregated bars around a backfill window must be verified
  for completeness before insert/update.

ISSUE P1-05
Severity: HIGH
Title: Live 1m volume and higher-timeframe volume are approximate and can be wrong

Current implementation:
- volume is refreshed only every 15 ticks,
- current 1m volume is copied from a REST candle snapshot only occasionally,
- 5m/15m volume is built from accumulators plus current 1m state.
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 10-11
  - src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 80-99
  - src/EthSignal.Infrastructure/Engine/LiveTickProcessor.cs lines 126-132

Why this is wrong:
- If the minute closes between volume refreshes, the final 1m volume may be
  stale or incomplete.
- Because higher-timeframe candles depend on that 1m volume, 5m/15m volume
  also diverges.

Impact:
- volume-based confirmation in SignalEngine can fail incorrectly,
- VWAP and volume analytics are degraded.

Required fix:
Option A (preferred):
- move to broker websocket/tick feed and build volume from actual trades if
  available.
Option B:
- refresh the current 1m broker candle every second and overwrite current
  1m volume with the latest value from the API.
- derive 5m and 15m strictly from closed 1m candles only.

Re-test requirement:
- compare finalized 5m volume against sum of 5 finalized 1m volumes.
- zero mismatches permitted.

ISSUE P1-06
Severity: HIGH
Title: Repeated gaps exist in 1m data and are not repaired or quarantined

Observed data issues:
- gap_events.csv shows 167 gap entries.
- Repeated daily gaps appear around 00:00-00:04.
- A large 121-minute gap also exists on 2026-03-21.

Why this matters:
- gaps invalidate aggregated bars crossing the missing region,
- indicators should not be treated as trustworthy across unresolved gaps.

Current implementation:
- gaps are detected and inserted into gap_events,
- but the pipeline continues using the affected data.

Required fix:
- Add a gap-aware validity layer.
- If a 5m or 15m candle spans a missing 1m slot:
  - either do not build it,
  - or mark it invalid and exclude it from indicator/signal generation.
- Add a repair/backfill recovery routine for unresolved gaps.

Re-test requirement:
- if any 1m gap exists inside a 5m/15m bucket, that bucket must not be
  used in indicator or signal computation.

------------------------------------------------------------

ISSUE P1-07
Severity: MEDIUM
Title: Gap detection marks the slot before the first fetched candle as missing

Current implementation:
- gap detection starts from floor(from).
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/GapDetector.cs lines 23-24

Why this can be misleading:
- If the requested period starts at 02:06:xx and the first actual candle is
  02:07, the detector marks 02:06 as missing even if the system never asked
  the broker for the full 02:06 minute.
- This is visible in gap_events.csv where the first expected gap is 02:06.

Required fix:
- start expected gap scanning from the first exact minute boundary that is
  logically expected to exist in the requested dataset,
- or explicitly classify this as "range_edge_partial" instead of
  "missing_candle".

Current implementation:

- Signals must continue using closed candles only.

- verify no provisional snapshot is stored in the final snapshot table unless
  explicitly intended and flagged.

PHASE 3 - 15M REGIME FILTER
====================================================================

ISSUE P3-01
Severity: HIGH
Title: Regime logic is stricter and narrower than the intended SRS

Current implementation:
- RegimeAnalyzer requires ALL of:
  - EMA20 > EMA50 / EMA20 < EMA50
  - EMA20 slope sign
  - Close vs VWAP
  - ADX >= 18
  - +DI > -DI or inverse
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/RegimeAnalyzer.cs lines 47-106

Mismatch with intended SRS:
- Intended filter:
  - EMA20 / EMA50 directional trend,
  - slope,
  - price vs VWAP,
  - ADX strength,
  - market structure context.
- Current implementation:
  - adds DI dominance as a mandatory condition,
  - does NOT include market structure logic (HH/HL or LH/LL),
  - returns only BULLISH / BEARISH if every condition is satisfied.

Impact:
- regime becomes NEUTRAL too often,
- signal throughput becomes lower than expected,
- the implementation is not the same strategy that was specified.

Required fix:
- Add explicit market-structure evaluation on 15m.
- Decide whether DI dominance is:
  - a mandatory rule,
  - a soft score component,
  - or removed.
- If the original strategy should be more permissive, move some conditions
  from mandatory to scored.

Recommended target:
- mandatory:
  - EMA20 relative to EMA50
  - EMA20 slope sign
  - ADX >= threshold
- scored/confirming:
  - close vs VWAP
  - DI dominance
  - HH/HL or LH/LL structure
  - volatility adequacy

Severity: HIGH
Title: Historical regime snapshots are not persisted across all 15m bars

Current implementation:
- Backfill stores only the latest classified regime, not the full regime

PHASE 4 - 5M SIGNAL ENGINE
====================================================================

ISSUE P4-01
Severity: HIGH
Title: Entry model does not follow the intended "next candle open after confirmation" behavior

Current implementation:
- Signal time = current 5m candle open time
- Entry price = current 5m close mid
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/SignalEngine.cs lines 192-202

Why this is wrong:
- The intended strategy was:
  - confirmation occurs on candle close,
  - entry is the next candle open (or explicit next-tick execution model).
- Current logic mixes candle-close confirmation with same-candle entry price.

Impact:
- backtest and live signal semantics are inconsistent,
- performance metrics can become overstated or ambiguous.

Required fix:
- Add explicit entry model:
  - signal_detected_at = just after candle close
  - planned_entry_time = next 5m candle open
  - planned_entry_price = next candle bid/ask open or next available live tick
- Persist both:
  - signal candle close
  - intended execution candle/time

ISSUE P4-02
Severity: HIGH
Title: Signal rules do not fully match the intended scalping setup

Current implementation:
- EMA condition checks only Close vs EMA20
- RSI buy zone allows 40-70 instead of a tighter recovery band
- RSI sell zone allows 30-60 instead of a tighter rejection band
- MACD condition uses hist positive OR improving
- Confirmation uses volume/VWAP/body ratio
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/SignalEngine.cs lines 54-188

Mismatch with intended SRS:
- Intended setup emphasized:
  - 15m directional bias
  - 5m pullback to EMA20 or VWAP zone
  - RSI recovery/rejection around a narrower zone
  - candle closes back in the trend direction
  - volume confirmation
- Current engine does not explicitly model:
  - pullback zone touch,
  - bounce/rejection from EMA20 or VWAP,
  - swing structure around the entry,
  - session/time filter,
  - nearby support/resistance veto.

Impact:
- strategy implemented in code is not the same as the target scalping logic.

Required fix:
- Replace simple "close vs EMA20" with a pullback-and-reclaim rule.
- Example BUY condition:
  1. prior bar or current bar touches/undercuts EMA20 or VWAP zone
  2. current close finishes back above EMA20
  3. RSI recovered upward from a predefined band
  4. MACD histogram improving
  5. 15m regime bullish
- Similar inverse logic for SELL.

ISSUE P4-03
Severity: HIGH
Title: Score threshold and mandatory gates may be too restrictive for practical throughput

Current implementation:
- Mandatory:
  - EMA
  - RSI
  - MACD
  - spread
- Threshold = 70
- Body confirmation is not scored, only gating.
- Relevant code:
  - src/EthSignal.Infrastructure/Engine/SignalEngine.cs lines 21-29
  - src/EthSignal.Infrastructure/Engine/SignalEngine.cs lines 165-188

Why this matters:
- If market is directional but ADX is slightly weak or volume is average,
  the score can stay below threshold even with a valid pullback setup.

Required fix:
- After Phase 1 data fixes, backtest the strategy over a larger sample.
- Measure signal frequency, win rate, and expectancy.
- Consider softening some gates:
  - move ADX to regime-only or scored-not-mandatory,
  - separate "hard fail" from "confidence reduction",
  - lower threshold only after empirical testing.

Title: signal_features table is never populated

  - ema20_minus_ema50

  - distance_to_vwap
  - volume_ratio
  - spread_pct
  - atr_pct
  - body_ratio
  - hour_of_day
  - regime_score
  - strategy_version
- Store into signal_features immediately after signal insert.

Re-test requirement:
- every inserted signal must have a complete signal_features row-set.

ISSUE P8-03
Severity: HIGH
Title: Authentication and session handling are not concurrency-safe

Current implementation:
- CapitalClient stores CST and security token in mutable shared fields.
- SendAuthorizedGetAsync can call AuthenticateAsync automatically.
- Dashboard/browser startup can create parallel requests while the
  background service is also authenticating.
- There is no lock/semaphore around AuthenticateAsync.
- Relevant code:
  - src/EthSignal.Infrastructure/Apis/CapitalClient.cs lines 18-19, 36-61, 166-195
  - src/EthSignal.Web/Program.cs lines 298-312

Observed behavior:
- The log shows:
  - a 429 on authentication,
  - then another authentication success shortly after.

Interpretation:
- Multiple concurrent auth/session attempts likely occurred.

Required fix:
- Protect session creation with a SemaphoreSlim.
- Store token expiry / session state.
- Never let dashboard endpoints trigger new auth while bootstrap auth is
  already in progress.
- Disable automatic browser opening in production and during automated tests.
