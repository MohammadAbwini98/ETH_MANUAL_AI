======================================================================
1. PURPOSE
======================================================================

This document defines the implementation of a machine learning (ML)
subsystem that learns from the existing decision audit trail, signal
outcomes, indicator snapshots, and regime classifications to:

1) Predict signal outcome probability (WIN/LOSS) before entry,
2) Dynamically recalibrate confidence scores beyond the current
   fixed-weight scoring model,
3) Adaptively tune strategy parameters in real-time based on
   detected market regime shifts and recent outcome patterns,
4) Increase signal generation frequency while maintaining or
   improving win rate and expectancy,
5) Identify optimal time-of-day, volatility, and micro-regime
   windows where signal quality is highest.

The current system uses a static weighted scoring model (8 components,
sum = 100) with fixed thresholds. The StrategyOptimizer (Phase 7) does
grid-search mutation of discrete parameter values. This SRS adds a
learned component that sits alongside (not replaces) the rule-based
engine, preserving auditability and rollback capability.

======================================================================
2. SCOPE
======================================================================

IN SCOPE:
  A. ML Feature Engineering Pipeline
     - Extract features from signal_decision_audit, indicator_snapshots,
       regime_snapshots, candles, and signal_outcomes tables.
     - Compute derived features not present in current indicators.

  B. Outcome Prediction Model (Binary Classifier)
     - Predict P(WIN) for a candidate signal before entry.
     - Filter or reweight signals based on predicted win probability.

  C. Confidence Recalibration Model
     - Replace or augment the fixed-weight scoring with a learned
       confidence model that outputs calibrated probabilities.

  D. Adaptive Parameter Tuner
     - Continuously adjust StrategyParameters based on rolling
       performance windows rather than offline optimizer runs.

  E. Signal Frequency Optimizer
     - Identify conditions where confidence thresholds can be safely
       lowered to capture more trades without degrading expectancy.

  F. Training & Validation Pipeline
     - Walk-forward training with embargo periods.
     - Integration with existing BacktestEngine and ReplayService.

  G. Runtime Integration
     - ML inference embedded in the live SignalEngine evaluation loop.
     - New domain models, DB tables, API endpoints, and safety guards.

OUT OF SCOPE:
  - Reinforcement learning (future phase).
  - Multi-symbol portfolio models.
  - Broker-side auto-order execution.
  - Deep learning / neural network architectures (kept to classical ML
    for interpretability and small-data suitability).
  - GPU infrastructure.

3. PROBLEM STATEMENT
======================================================================

3.1 Current Limitations

The current SignalEngine.EvaluateWithDecision() uses fixed weights:
  WeightRegime=20, WeightPullback=20, WeightRsi=15, WeightMacd=15,
  WeightAdx=10, WeightVolume=10, WeightSpread=5, WeightBody=5

These weights are identical for all market conditions. The existing
StrategyOptimizer does grid-search over discrete parameter values but:
  - Explores only ~10 parameters from a fixed grid (adxValues,
    volumeValues, stopAtrValues, etc. in OptimizerService.cs:267-275).
  - Uses a linear composite score function (OptimizerService.cs:224-234).
  - Cannot capture nonlinear interactions between features.
  - Cannot adapt to time-varying market microstructure.

Result: The system generates signals at a relatively low density
(~1-3 per hour on 5m candles) because the fixed ConfidenceBuyThreshold
and ConfidenceSellThreshold (both 65) are conservative across all
conditions. In high-conviction environments (strong trend + high ADX +
aligned MACD), the threshold could be lowered safely. In choppy
environments, it should be raised.

3.2 Available Training Data

The PostgreSQL database contains rich labeled data:

  Table                     | Records | Key Fields
  --------------------------+---------+------------------------------
  signal_decision_audit     | ~588K   | All indicators, regime, reason
                            |         | codes, confidence, outcome
  signals                   | ~78K    | Direction, entry/TP/SL, score
  signal_outcomes           | ~26K    | WIN/LOSS/EXPIRED, PnlR, MFE/MAE
  indicator_snapshots       | ~1.6M   | 14 indicator values per bar
  regime_snapshots          | ~202K   | BULLISH/BEARISH/NEUTRAL + score
  candles_5m                | ~622K   | Full OHLCV with bid/ask/mid
  strategy_parameter_sets   | ~10     | Versioned parameter snapshots

The signal_decision_audit table is the richest source: it stores the
full indicator snapshot, regime used, reason codes, and outcome for
EVERY 5m evaluation (including NO_TRADE decisions). This gives us both
positive labels (signals that were taken and resolved) and implicit
negative labels (decisions where no trade was taken but market moved
favorably — missed opportunities).

3.3 Why ML Over Extended Grid Search

  - Feature interactions: RSI+ADX+HourOfDay jointly predict better
    than each feature independently weighted.
  - Regime-conditional weights: A pullback in a high-ADX trending
    market should be weighted differently than in a low-ADX range.
  - Temporal patterns: Certain hours/sessions produce systematically
    better signals (London/NY overlap vs Asia session for ETH).
  - Continuous optimization: ML models retrain on rolling windows,
    adapting faster than periodic optimizer runs.
  - Confidence calibration: An ML model can output P(WIN)=0.72 instead
    of "score=80", enabling threshold optimization based on expected
    value rather than arbitrary cutoffs.

4.2 Design Principles

  DP-1: AUGMENT, NOT REPLACE
    The ML layer augments the rule-based engine. The original
    SignalEngine score is always computed and logged. The ML model
    provides an additional confidence signal. The final decision uses
    both, with the rule-based engine as the floor.

  DP-2: INTERPRETABLE MODELS
    Use gradient-boosted trees (LightGBM/XGBoost equivalent) and
    logistic regression — not black-box neural nets. Feature importance
    is logged and auditable.

  DP-3: SAFETY-FIRST ACTIVATION
    ML predictions are logged in shadow mode before influencing trades.
    Activation requires explicit operator approval after validation
    period.

  DP-4: CAUSALLY CORRECT TRAINING
    Training data must respect temporal ordering. No future data
    leakage. Walk-forward validation with embargo gaps.

  DP-5: GRACEFUL DEGRADATION
    If the ML model is unavailable, stale, or underperforming, the
    system falls back to the rule-based engine with no disruption.

  DP-6: C# NATIVE
    All ML inference runs in-process in C#. Training can use Python
    offline, but the production model is loaded as a serialized model
    file or ONNX format for C# inference via ML.NET or ONNX Runtime.

5. FEATURE ENGINEERING
======================================================================

5.1 Feature Categories

The ML model operates on a feature vector built at signal evaluation
time. Features are grouped into categories:

  CATEGORY A: Current Indicator Values (14 features)
  ─────────────────────────────────────────────────
  Extracted directly from IndicatorSnapshot at evaluation time.
  These are the same values stored in signal_decision_audit.

  Feature                | Source                    | Type
  -----------------------+---------------------------+-------
  ema20                  | snap.Ema20                | float
  ema50                  | snap.Ema50                | float
  rsi14                  | snap.Rsi14                | float
  macd_hist              | snap.MacdHist             | float
  adx14                  | snap.Adx14                | float
  plus_di                | snap.PlusDi               | float
  minus_di               | snap.MinusDi              | float
  atr14                  | snap.Atr14                | float
  vwap                   | snap.Vwap                 | float
  volume_sma20           | snap.VolumeSma20          | float
  spread                 | snap.Spread               | float
  close_mid              | snap.CloseMid             | float
  volume                 | candle.Volume             | float
  body_ratio             | ComputeBodyRatio(candle)  | float

  CATEGORY B: Derived/Relational Features (18 features)
  ─────────────────────────────────────────────────────
  Computed from current + historical indicator values.

  Feature                    | Computation                         | Rationale
  ---------------------------+-------------------------------------+------------------
  ema20_minus_ema50          | ema20 - ema50                       | Trend strength
  ema20_minus_ema50_pct      | (ema20-ema50)/close_mid             | Normalized trend
  ema20_slope_3              | (ema20[t]-ema20[t-3])/3             | Trend momentum
  ema20_slope_5              | (ema20[t]-ema20[t-5])/5             | Trend momentum (longer)
  rsi14_delta                | rsi14[t] - rsi14[t-1]              | RSI momentum
  rsi14_delta_3              | rsi14[t] - rsi14[t-3]              | RSI momentum (longer)
  macd_hist_delta            | macd_hist[t] - macd_hist[t-1]      | MACD acceleration
  macd_hist_delta_3          | macd_hist[t] - macd_hist[t-3]      | MACD acceleration (longer)
  adx14_delta                | adx14[t] - adx14[t-1]              | Trend strengthening
  atr14_pct                  | atr14 / close_mid                   | Normalized volatility
  atr14_delta_pct            | (atr14[t]-atr14[t-1])/atr14[t-1]  | Vol expansion/contraction
  distance_to_ema20_pct      | (close_mid - ema20) / close_mid     | Pullback depth
  distance_to_vwap_pct       | (close_mid - vwap) / close_mid      | VWAP deviation
  volume_ratio               | volume / volume_sma20               | Relative volume
  spread_pct                 | spread / close_mid                  | Normalized spread
  di_differential            | plus_di - minus_di                  | Directional pressure
  di_ratio                   | plus_di / max(minus_di, 0.01)      | Directional dominance
  candle_range_pct           | (high - low) / close_mid            | Bar range vs price

  CATEGORY C: Contextual Features (12 features)
  ─────────────────────────────────────────────
  Market context and temporal information.

  Feature                    | Source                              | Rationale
  ---------------------------+-------------------------------------+------------------
  regime_label               | regime.Regime (encoded 0/1/2)      | Market state
  regime_score               | regime.RegimeScore                  | Regime confidence
  regime_age_bars            | bars since regime classification    | Regime freshness
  rule_based_score           | SignalEngine confidence score       | Original score
  direction_encoded          | BUY=1, SELL=-1                     | Trade direction
  hour_of_day                | signal_time.Hour (0-23)            | Session effects
  day_of_week                | signal_time.DayOfWeek (0-6)        | Day patterns
  minutes_since_open         | minutes since 00:00 UTC            | Session timing
  is_london_session          | 07:00-16:00 UTC flag               | Session flag
  is_ny_session              | 13:00-21:00 UTC flag               | Session flag
  is_asia_session            | 23:00-07:00 UTC flag               | Session flag
  is_overlap                 | London+NY overlap flag             | High liquidity flag

  CATEGORY D: Lookback Pattern Features (14 features)
  ──────────────────────────────────────────────────
  Patterns over recent N bars (lookback window).

  Feature                    | Window | Computation
  ---------------------------+--------+----------------------------
  recent_win_rate_10         | 10 sig | Wins/Resolved over last 10 signals
  recent_win_rate_20         | 20 sig | Wins/Resolved over last 20 signals
  recent_avg_pnl_r_10        | 10 sig | Average PnlR over last 10 signals
  recent_avg_pnl_r_20        | 20 sig | Average PnlR over last 20 signals
  consecutive_wins           | recent | Current win streak length
  consecutive_losses         | recent | Current loss streak length
  bars_since_last_signal     | bars   | Signal density indicator
  avg_atr_20_bars            | 20 bar | Rolling volatility baseline
  atr_zscore                 | 20 bar | (current_atr - mean) / std
  avg_volume_10_bars         | 10 bar | Recent volume trend
  volume_zscore              | 10 bar | (current_vol - mean) / std
  price_range_20_bars_pct    | 20 bar | (max-min)/mid over 20 bars
  regime_changes_last_20     | 20 bar | Regime stability indicator
  pullback_depth_pct         | 5 bar  | Max distance from EMA20 in window

  TOTAL FEATURE VECTOR: 58 features

5.2 Feature Storage

Features are computed at evaluation time and stored in a new table
(see Section 10) for both training and audit purposes.

5.3 Feature Normalization

  - Tree-based models (primary): No normalization needed.
  - Logistic regression (secondary): StandardScaler applied.
  - Normalization parameters (mean/std) are computed from the training
    set and stored with the model artifact.

5.4 Feature Selection Protocol

After initial model training:
  1. Compute SHAP values for feature importance ranking.
  2. Remove features with importance < 1% of top feature.
  3. Check for multicollinearity (VIF > 10 → remove redundant).
  4. Re-train with reduced feature set.
  5. Log selected features in model metadata for reproducibility.

======================================================================
6. ML MODELS
======================================================================

6.1 Model M1: Outcome Predictor (Primary)

  Purpose: Predict P(WIN) for a candidate signal.
  Type:    Gradient Boosted Decision Trees (GBDT)
  Target:  Binary — WIN=1, LOSS/EXPIRED/AMBIGUOUS=0
  Input:   58-feature vector (Section 5)
  Output:  P(WIN) in [0.0, 1.0]

  Hyperparameters (initial, subject to cross-validation):
    - n_estimators:     200-500 (early stopping on validation)
    - max_depth:        4-6 (prevent overfitting on small dataset)
    - learning_rate:    0.05-0.1
    - min_child_weight: 20 (minimum leaf size, prevents overfitting)
    - subsample:        0.8 (row sampling per tree)
    - colsample_bytree: 0.8 (feature sampling per tree)
    - reg_alpha:        0.1 (L1 regularization)
    - reg_lambda:       1.0 (L2 regularization)
    - scale_pos_weight: auto (class imbalance correction)

  Training regime:
    - Walk-forward with 3-5 folds (aligned with OptimizerService).
    - Embargo gap: 60 bars (5 hours) between train/validation sets
      to prevent label leakage from overlapping outcome windows.
    - Minimum training set: 200 resolved signals.
    - Retrain trigger: every 100 new resolved signals OR weekly.

  Evaluation metrics (logged per fold):
    - AUC-ROC (primary: discrimination ability)
    - Brier Score (calibration quality)
    - Precision at 50% recall (minimum useful recall)
    - Expected Calibration Error (ECE)
    - Log loss

  Acceptance criteria for production deployment:
    - AUC-ROC > 0.55 on out-of-sample validation set
    - Brier Score < 0.25
    - ECE < 0.05
    - No fold with AUC-ROC < 0.52

6.2 Model M2: Confidence Recalibrator

  Purpose: Map (rule_based_score, ml_pwin, market_context) to
           calibrated confidence in [0, 100].
  Type:    Isotonic Regression (Platt scaling as fallback)
  Input:   rule_based_score + M1 P(WIN) + regime_score
  Output:  Calibrated confidence score (0-100)

  Training: Fit on validation fold of M1.
  Purpose:  Ensure that "confidence=70" means roughly 70% historical
            win rate — currently the rule-based score is not calibrated.

6.3 Model M3: Dynamic Threshold Model

  Purpose: Determine optimal ConfidenceThreshold for current market
           conditions to maximize expectancy while maintaining minimum
           signal frequency.
  Type:    Lookup table / simple regression
  Input:   regime_label, adx14, atr14_pct, hour_of_day, recent_win_rate
  Output:  Recommended ConfidenceBuyThreshold, ConfidenceSellThreshold

  Method:
    1. Bin historical signals by (regime, ADX bucket, ATR bucket,
       session).
    2. For each bin, compute the optimal threshold that maximizes:
         Objective = Expectancy * sqrt(TradeCount)
       This rewards both quality (expectancy) and quantity (frequency).
    3. Store as a threshold lookup table.
    4. At runtime, lookup the threshold for current conditions.

  Constraints:
    - Threshold must be in [40, 90] (never below 40 for safety).
    - If a bin has fewer than 20 signals, fall back to the global
      default (65).
    - Maximum threshold change between consecutive evaluations: +-5
      (prevents whiplash).

  This model directly addresses the "high signal frequency" goal:
  in well-characterized favorable conditions, it lowers the threshold
  to capture more signals; in adverse conditions, it raises it.

6.4 Model M4: Regime Enhancement Predictor (Optional - Phase 2)

  Purpose: Predict regime BEFORE 15m bar closes, using 5m features
           to get earlier regime signals.
  Type:    Multi-class classifier (BULLISH/BEARISH/NEUTRAL)
  Input:   Last 3 bars of 5m indicators + partial 15m bar features
  Output:  Predicted regime + confidence

  This is Phase 2 because the current RegimeAnalyzer on 15m bars works
  reliably. The ML regime predictor would provide a "leading indicator"
  of regime changes, allowing earlier signal generation.

7. TRAINING PIPELINE
======================================================================

7.1 Data Extraction

  The training pipeline extracts labeled data from PostgreSQL:

  Step 1: Query resolved signals with outcomes
  ───────────────────────────────────────────
  SELECT
    s.signal_id, s.direction, s.confidence_score, s.signal_time_utc,
    s.entry_price, s.tp_price, s.sl_price, s.regime,
    o.outcome_label, o.pnl_r, o.mfe_r, o.mae_r, o.bars_observed
  FROM "ETH".signals s
  JOIN "ETH".signal_outcomes o ON s.signal_id = o.signal_id
  WHERE o.outcome_label IN ('WIN', 'LOSS', 'EXPIRED', 'AMBIGUOUS')
  ORDER BY s.signal_time_utc

  Step 2: Join with indicator snapshots at signal time
  ────────────────────────────────────────────────────
  For each signal, retrieve the IndicatorSnapshot at
  s.signal_time_utc from indicator_snapshots.

  Step 3: Join with regime snapshots
  ──────────────────────────────────
  For each signal, retrieve the most recent RegimeResult at or
  before the signal time.

  Step 4: Compute derived features (Category B, C, D)
  ───────────────────────────────────────────────────
  Using lookback windows over indicator_snapshots and prior signals.

  Step 5: Construct feature matrix X and label vector y
  ────────────────────────────────────────────────────
  y = 1 if outcome_label == WIN, else 0

7.2 Walk-Forward Training Protocol

  Unlike standard cross-validation, financial data requires temporal
  ordering. The protocol:

  Given data from time T0 to TN:

  Fold 1:
    Train: [T0, T1)     Embargo: [T1, T1+embargo)    Val: [T1+embargo, T2)
  Fold 2:
    Train: [T0, T2)     Embargo: [T2, T2+embargo)    Val: [T2+embargo, T3)
  Fold 3:
    Train: [T0, T3)     Embargo: [T3, T3+embargo)    Val: [T3+embargo, T4)
  ...

  Where:
    - embargo = OutcomeTimeoutBars * TimeframePrimary minutes
              = 60 * 5 = 300 minutes (5 hours)
    - Fold boundaries aligned with HistoricalReplayService date ranges.
    - Expanding window (cumulative training data) is preferred over
      sliding window for small datasets.

  Alignment with OptimizerService:
    The ML training uses the same fold structure as
    OptimizerService.GenerateFolds() for consistency. The embargo gap
    is additional — the optimizer does not currently use one, but SHOULD
    (see Section 13, recommendation R-02).

7.3 Retraining Schedule

  Trigger conditions (any of):
    T1. 100 new resolved signals since last training.
    T2. 7 calendar days since last training.
    T3. Rolling 20-signal win rate deviates > 15% from model's
        predicted average (concept drift detection).
    T4. Manual trigger via API endpoint.

  Retraining process:
    1. Extract latest dataset (Step 7.1).
    2. Walk-forward train (Step 7.2).
    3. Evaluate acceptance criteria (Section 6.1).
    4. If accepted: save model as CANDIDATE.
    5. Run shadow evaluation for 50 signals.
    6. If shadow metrics are within acceptance: promote to ACTIVE.
    7. Log all artifacts (model file, metrics, feature importance).

7.4 Training Infrastructure

  Option A (Recommended): Python Training + ONNX Export
  ──────────────────────────────────────────────────────
    - Training script in Python (scikit-learn, LightGBM, XGBoost).
    - Export trained model as ONNX format.
    - C# runtime loads ONNX model via Microsoft.ML.OnnxRuntime NuGet.
    - Training triggered via a CLI tool or scheduled job.
    - Model file stored at: /models/{model_type}_{version}.onnx
    - Model metadata (features, metrics, training range) stored in DB.

  Option B (Alternative): ML.NET Native
  ──────────────────────────────────────
    - Training and inference both in C# via ML.NET.
    - Uses Microsoft.ML.LightGbm trainer.
    - Simpler deployment (single-language), but less flexible.
    - Model file stored at: /models/{model_type}_{version}.zip

  For both options, the inference interface is identical (see Section 8).

7.5 Concept Drift Detection

  The model monitors for distribution shift:

  Metric                  | Window | Threshold | Action
  ------------------------+--------+-----------+------------------
  Rolling AUC-ROC         | 50 sig | < 0.52    | Alert + retrain
  Rolling Brier Score     | 50 sig | > 0.28    | Alert + retrain
  Prediction mean shift   | 50 sig | > 0.15    | Alert + log
  Feature distribution    | 100 bar| KS > 0.20 | Log warning
  Win rate vs predicted   | 30 sig | > 15% gap | Trigger retrain

  When drift is detected:
    1. Log warning with drift metrics.
    2. If model is ACTIVE: fall back to rule-based scoring.
    3. Trigger retraining pipeline.
    4. After retrain + validation: re-activate if criteria met.

8.2 MlPrediction (EthSignal.Domain/Models/MlPrediction.cs)

  public sealed record MlPrediction
  {
      public required Guid PredictionId { get; init; }
      public required Guid EvaluationId { get; init; }
      public required string ModelVersion { get; init; }
      public required string ModelType { get; init; }

      /// P(WIN) from outcome predictor (0.0 to 1.0).
      public decimal PredictedWinProbability { get; init; }

      /// Calibrated confidence score (0-100).
      public int CalibratedConfidence { get; init; }

      /// Dynamic threshold for current conditions.
      public int RecommendedThreshold { get; init; }

      /// Expected value: P(WIN)*avgWinR - P(LOSS)*avgLossR
      public decimal ExpectedValueR { get; init; }

      /// Inference latency in microseconds.
      public int InferenceLatencyUs { get; init; }

      /// Whether this prediction influenced the trade decision.
      public bool IsActive { get; init; }

      /// Mode: SHADOW (log only) or ACTIVE (influences decision).

9.3 Modified SignalEngine Flow

  The existing SignalEngine.EvaluateWithDecision() is extended with an
  optional ML enhancement step. The modification is minimal:

  CURRENT (lines 42-205 of SignalEngine.cs):
    1. Check context/regime → NO_TRADE if missing
    2. Score BUY and SELL independently
    3. Check spread gate
    4. Pick higher-scoring direction
    5. Check entry condition (pullback + momentum)
    6. Check score threshold
    7. Return signal or NO_TRADE

  ENHANCED (new flow — changes in steps 6-7 only):
    1-5. UNCHANGED (rule-based engine runs exactly as before)

    6. [NEW] If MlMode != DISABLED and MlInferenceService.IsReady:
       a. Build MlFeatureVector from current state.
       b. Call MlInferenceService.Predict(features).
       c. Compute blended confidence:
          blended = (1 - blendWeight) * ruleBasedScore
                  + blendWeight * (mlPrediction.CalibratedConfidence)
       d. Determine effective threshold:
          - If MlDynamicThresholdsEnabled:
            threshold = mlPrediction.RecommendedThreshold
          - Else: threshold = p.ConfidenceBuyThreshold (unchanged)
       e. Store MlPrediction in SignalDecision for audit.

    7. Check blended score vs effective threshold:
       - If MlMode == SHADOW:
         Use original rule-based score and threshold (unchanged).
         Log ML prediction for comparison only.
       - If MlMode == ACTIVE:
         Use blended score and effective threshold.
         Additional gate: mlPrediction.PredictedWinProbability
         must be >= MlMinWinProbability (default 0.50).

    8. UNCHANGED: Return signal or NO_TRADE.

  Key guarantee: When MlMode == DISABLED or model is unavailable,
  the engine behaves IDENTICALLY to the current implementation.
  No code path in the existing engine is modified — only new code
  is added after step 5.

12. SIGNAL FREQUENCY ENHANCEMENT STRATEGY
======================================================================

This section addresses the primary goal: increasing signal generation
frequency while maintaining quality.

12.1 Current Signal Density Analysis

  Based on the exported data:
  - 5m candles: ~892 over ~3 days = ~297/day = ~12.4/hour
  - Signals generated: ~78K entries total
  - With ConfidenceThreshold=65 and mandatory gates, only a small
    fraction of 5m evaluations produce actionable signals.

  Primary blocking reasons (from signal_decision_audit):
    1. REGIME_NEUTRAL          — Most common, blocks all entries
    2. PULLBACK_NOT_VALID      — No pullback pattern detected
    3. NO_ENTRY_CONDITION      — Pullback present but no momentum
    4. SCORE_BELOW_THRESHOLD   — Score 50-64, just below threshold
    5. MACD_CONFIRMATION_FAILED — MACD histogram didn't cross zero

12.2 ML-Driven Frequency Improvements

  APPROACH 1: Dynamic Thresholds (Model M3)
  ──────────────────────────────────────────
  Current: Fixed threshold 65 for all conditions.
  Enhanced: Threshold varies by market context.

  Example (from historical data analysis):
    Context                            | Threshold | Rationale
    -----------------------------------+-----------+-------------------
    BULLISH + ADX>30 + London session  | 50        | Strong trend, high win rate
    BULLISH + ADX 20-30 + any session  | 60        | Moderate trend
    NEUTRAL + ADX<20 + Asia session    | 80        | Choppy, low liquidity
    BEARISH + ADX>25 + NY session      | 55        | Clear downtrend
    Any + recent_win_rate_20 > 60%     | -5 adj.   | Hot streak bonus
    Any + consecutive_losses >= 3      | +10 adj.  | Cooling-off raise

  Impact: Signals scoring 50-64 (currently blocked) become actionable
  in favorable conditions. Expected frequency increase: +30-60%.

  APPROACH 2: Relaxed Mandatory Gates with ML Backup
  ──────────────────────────────────────────────────
  Current: pullback AND (RSI OR MACD) is mandatory.
  Enhanced: If ML P(WIN) > 0.65, allow entries where:
    - Pullback is present but only RSI/MACD is missing (not both).
    - ADX is slightly below threshold (15-19 vs required 20).
    - Body ratio is below 0.2 but direction matches.

  This is controlled by a new parameter:
    MlOverrideMandatoryGates: bool (default false)
  And requires MlMode == ACTIVE + P(WIN) > MlMinWinProbability + 0.15.

  Safety: The ML model must have demonstrated AUC > 0.60 and the
  specific gate being relaxed must have historical evidence that
  signals passing all other gates but failing this one still have
  positive expectancy.

  APPROACH 3: Neutral Regime Entry with ML Gating
  ───────────────────────────────────────────────
  Current: NeutralRegimePolicy = BlockAllEntriesInNeutral
  Enhanced: Allow entries in NEUTRAL regime ONLY when ML P(WIN) > 0.60
  and the regime_score >= 3 (near-directional).

  This is a natural extension of the existing
  NeutralRegimePolicy enum. A new value:
    AllowMlGatedEntriesInNeutral
  enables entries in NEUTRAL only when the ML model is confident.

  Expected impact: +20-40% more signals, since REGIME_NEUTRAL is the
  #1 blocking reason.

  APPROACH 4: Multi-Timeframe Signal Aggregation
  ─────────────────────────────────────────────
  Current: Signals evaluated only on 5m candles.
  Enhanced: ML model aggregates features from 5m + 15m + 1h snapshots
  for a richer context, potentially identifying setups visible on
  multiple timeframes simultaneously.

  Additional features:
    - alignment_5m_15m: Do 5m and 15m indicators agree on direction?
    - alignment_5m_1h:  Do 5m and 1h indicators agree?
    - mtf_score: Weighted average of direction signals across TFs.

  Signals with multi-timeframe alignment and ML P(WIN) > 0.55 can
  use a lower confidence threshold.

12.3 Frequency Target

  Current estimated density: ~1-3 signals/hour (with many hours at 0).
  Target density: ~3-8 signals/hour in trending conditions.
  Constraint: Expectancy (win_rate * avg_win - loss_rate * avg_loss)
              must remain > 0 on rolling 50-signal windows.

  The frequency increase comes from:
    - Lower thresholds in favorable conditions: +30-60%
    - Neutral regime entries with ML gating:    +20-40%
    - Relaxed mandatory gates with ML backup:   +10-20%
    - Total estimated increase:                 +60-120%

13. ADAPTIVE PARAMETER TUNER
======================================================================

13.1 Purpose

  The current OptimizerService runs offline, generates candidate
  parameter sets via grid search, and requires manual activation.
  The Adaptive Parameter Tuner provides continuous, fine-grained
  parameter adjustments based on recent performance.

13.2 Parameters Subject to Adaptive Tuning

  Only a subset of StrategyParameters are adaptively tuned.
  Risk management parameters are NEVER auto-tuned (safety constraint).

  Tunable Parameters:
    - ConfidenceBuyThreshold   (range: 40-90)
    - ConfidenceSellThreshold  (range: 40-90)
    - WeightRegime             (range: 10-30)
    - WeightPullback           (range: 10-30)
    - WeightRsi                (range: 5-25)
    - WeightMacd               (range: 5-25)
    - WeightAdx                (range: 5-20)
    - WeightVolume             (range: 5-20)
    - WeightSpread             (range: 0-10)
    - WeightBody               (range: 0-10)
    - PullbackZonePct          (range: 0.002-0.010)
    - VolumeMultiplierMin      (range: 0.4-1.5)
    - AdxTrendThreshold        (range: 15-30)

  NEVER auto-tuned (require manual activation):
    - AccountBalanceUsd
    - RiskPerTradePercent
    - HardMaxRiskPercent
    - DailyLossCapPercent
    - MaxConsecutiveLossesPerDay
    - StopAtrMultiplier
    - TargetRMultiple

13.3 Tuning Algorithm

  The tuner uses a Bayesian-inspired approach:

  1. Maintain a rolling performance window of last N resolved signals
     (N = 50 by default).
  2. Compute performance metrics for the window:
     win_rate, avg_pnl_r, profit_factor, expectancy.
  3. Compare to baseline expectancy (from initial parameter set).
  4. If performance is degrading (expectancy < 80% of baseline):
     a. Analyze which score components contributed to losing signals.
     b. Increase weight of components that discriminated best between
        wins and losses (using feature importance from M1 model).
     c. Raise confidence threshold by +5 (reduce frequency for quality).
  5. If performance is strong (expectancy > 120% of baseline):
     a. Lower confidence threshold by -3 (increase frequency).
     b. Slightly widen PullbackZonePct (catch more pullbacks).
  6. Constraint: total weight sum must remain 100 (rescale if needed).
  7. Constraint: change delta per adjustment <= 20% of current value.

  Adjustment frequency: Once per 25 resolved signals (not per bar).

13.4 Safety Guards

  G1: All adaptive changes are logged as a new StrategyParameterSet
      with status = "adaptive" and a pointer to the parent set.
  G2: If any rolling 30-signal window has expectancy < -0.1R,
      immediately revert to last known good parameter set.
  G3: Maximum 3 adaptive adjustments before requiring a full
      optimizer run for validation.
  G4: Adaptive tuning is disabled during the first 100 signals of
      a new parameter set activation (learning period).
  G5: Operator can disable adaptive tuning via API or parameter flag.

         - Gradually increase to 0.5 over 1-2 weeks
         - Monitor expectancy throughout
         - Effort: Operations (not code)

  P6-03: Performance optimization
         - Inference latency profiling (target: < 1ms per prediction)
         - Memory footprint optimization
         - Model size optimization (pruning, quantization)
         - Effort: Small

  P6-04: Documentation and runbooks
         - Training runbook
         - Model activation/rollback procedures
         - Drift response playbook

15.3 Backtesting Validation

  The ML system MUST be validated against historical data before
  live deployment:

  1. Train model on data up to T-30 days.
  2. Run replay from T-30 to T with ML enhancement.
  3. Compare:
     a. Rule-based signals: count, win rate, expectancy
     b. ML-enhanced signals: count, win rate, expectancy
     c. ML must show improvement in at least ONE of:
        - Higher expectancy (>= +10%)
        - Higher signal frequency with maintained expectancy
        - Lower max drawdown with maintained expectancy
  4. Results logged and presented for operator review.

15.4 Shadow Mode Protocol

  Before ACTIVE mode is allowed:

  1. Run ML in SHADOW mode for minimum 50 live signals.
  2. For each signal, record:
     - Would ML have changed the decision? (yes/no)
     - If yes: what would the outcome have been?
  3. Compute "shadow metrics":
     - Shadow win rate vs actual win rate
     - Shadow expectancy vs actual expectancy
     - Signals that ML would have added vs removed
  4. Require: shadow expectancy >= 90% of actual expectancy.
  5. Require: no signal removed by ML that turned out to be a WIN
     with PnlR > 2.0 (i.e., ML must not filter out big winners).

16. SAFETY & ROLLBACK
======================================================================

16.1 Kill Switch

  The ML system has a multi-level kill switch:

  Level 1: Set MlMode = DISABLED in StrategyParameters.
           Effect: Immediate. ML inference stops. Pure rule-based.
           Method: API POST /api/admin/ml/mode { mode: "DISABLED" }

  Level 2: Retire active model.
           Effect: MlInferenceService.IsReady becomes false.
           Method: API POST /api/admin/ml/models/{id}/retire

  Level 3: Automatic revert on performance breach.
           Trigger: Rolling 30-signal expectancy < -0.1R.
           Effect: MlMode set to DISABLED, alert logged.

16.2 Rollback Procedure

  1. Set MlMode = DISABLED (instant, no restart needed).
  2. Verify system is operating in rule-based mode (check /health).
  3. Investigate cause of poor performance.
  4. If ML model issue: retire model, retrain with more data.
  5. If integration issue: fix code, re-validate in shadow mode.
  6. Re-enable via graduated activation (Section 14.6).

16.3 Audit Trail

  Every ML prediction is persisted in ml_predictions table with:
  - Model version used
  - Full feature vector reference (via evaluation_id)
  - Prediction values
  - Whether prediction was active or shadow
  - Inference latency

  This enables:
  - Post-mortem analysis of any ML-influenced trade
  - A/B comparison of ML vs rule-based decisions
  - Regulatory compliance (if applicable)

17. PERFORMANCE REQUIREMENTS
======================================================================

  P-01: ML inference latency must be < 1ms (p99) per prediction.
        The 5m candle processing has a 5-minute window, so this is
        not a tight constraint, but fast inference prevents bottlenecks
        during historical replay.

  P-02: Model loading must complete within 5 seconds at startup.

  P-03: Feature extraction must be < 500us per evaluation.

  P-04: Memory overhead of loaded model must be < 50MB.

  P-05: Training pipeline must complete within 30 minutes for
        1000 training samples with 5-fold walk-forward validation.

  P-06: Drift detection checks must run within 10ms per evaluation.

  P-07: Database writes for ML predictions must not block the signal
        evaluation pipeline (async fire-and-forget with retry).

      (only ~78K signals, ~26K      |          | embargo gaps, conservative
      outcomes)                     |          | hyperparameters, minimum
                                    |          | 200-sample training set
                                    |          |

      changes invalidate model)     |          | automatic retraining,
                                    |          | fallback to rule-based
                                    |          |
  R3: Feature leakage (future data  | Critical | Strict temporal ordering in
      leaks into training)          |          | training, embargo gaps,
                                    |          | code review checklist
                                    |          |
  R4: ML increases frequency but    | Medium   | Expectancy constraint on
      degrades quality              |          | rolling windows, safety
                                    |          | guards, gradual activation
                                    |          |
  R5: Inference latency impacts     | Low      | ONNX Runtime is fast,
      live processing               |          | < 1ms target, async fallback
                                    |          |
  R6: Model file corruption or      | Medium   | Checksum validation on load,
      incompatibility               |          | feature version tracking,
                                    |          | graceful fallback
                                    |          |
  R7: Adaptive tuner oscillates     | Medium   | Max 3 adjustments before
      parameters (whiplash)         |          | requiring full optimizer,
                                    |          | max delta constraint,
                                    |          | cooling-off period

21. SUCCESS METRICS
======================================================================

  The ML enhancement is considered successful if, after Phase 6
  validation, the following metrics improve vs rule-based baseline:

  Metric                          | Target         | Measurement
  --------------------------------+----------------+------------------
  Signal frequency (signals/hour) | +50% or more   | Live, 7-day avg
  Expectancy per signal           | >= baseline     | Live, 30-day avg
  Win rate                        | >= baseline -5% | Live, 30-day avg
  Profit factor                   | >= baseline     | Live, 30-day avg
  Max drawdown                    | <= baseline+10% | Live, 30-day worst
  ML model AUC-ROC                | > 0.55         | Walk-forward val
  Confidence calibration (ECE)    | < 0.05         | Walk-forward val
  Inference latency (p99)         | < 1ms          | Live measurement
  System uptime impact            | 0%             | Live monitoring

  Primary success criterion: The product of (signal_frequency *
  expectancy_per_signal) increases by >= 30%. This captures both
  frequency and quality in a single metric.

22. RECOMMENDATIONS FOR EXISTING CODE

  These are not part of the ML implementation but are improvements

  R-01: Add embargo gap to OptimizerService walk-forward folds.
