ISSUE U-02
Severity: CRITICAL
Title: Startup destroys all data on every run
Location:
- ETH_G/Program.cs lines 25-30
- ETH_G/Db/DbInitializer.cs lines 24-54
Observed behavior:
Program.cs calls:
- InitializeAsync()
- TruncateAllAsync()
DbInitializer.InitializeAsync() drops every timeframe table and recreates it.
Then TruncateAllAsync() clears all rows again.
Why this is wrong:
This directly violates the earlier audit and the main SRS requirement that history, signals, outcomes, and learning data persist across runs.
A trading system cannot learn, validate, backtest, or audit results if it destroys all state at startup.
Required behavior:
Startup must be idempotent and non-destructive.
Only create missing schema objects.
Never drop or truncate production data automatically.
Required fix:
- Remove Program.cs lines 28-29 in production startup path.
- Replace DbInitializer.InitializeAsync() with migration logic that uses CREATE TABLE IF NOT EXISTS and ALTER TABLE migration steps only.
- Move any destructive operation behind an explicit manual admin command or separate maintenance script.
Acceptance gate:
A restart must preserve existing candle rows and all higher-level tables.

ISSUE U-04
Severity: CRITICAL
Title: Live rollover still closes candles with the first price of the next bucket
Location:
- ETH_G/Engine/CandleEngine.cs lines 149-159
Observed behavior:
When a boundary is crossed, the previous candle is finalized using the CURRENT price variable:
- High = max(current.High, price)
- Low = min(current.Low, price)
- Close = price
But price is already the first tick seen after the bucket boundary.
Why this is wrong:
This reproduces the exact rollover defect from the first audit.
The close of candle N becomes the first tick of candle N+1.
This corrupts OHLC, indicators, higher-timeframe aggregates, and any future signals/backtests.
Required behavior:
The previous candle must close using the LAST price observed before the boundary.
Required fix:
Store previous spot/tick and use it to finalize the previous bucket.
Pseudo-fix:
1. Keep previousPrice and previousTimestamp.
2. On boundary crossing:
   finalClose = previousPrice
3. Open the new candle with currentPrice.
Acceptance gate:
For a boundary-crossing sequence [price(t-1), price(t boundary)], candle N closes at price(t-1) and candle N+1 opens at price(t boundary).

ISSUE U-05
Severity: CRITICAL
Title: Higher timeframe live candles are built independently from tick stream instead of being derived from closed 1m bars
Location:
- ETH_G/Engine/CandleEngine.cs lines 144-199
Observed behavior:
The live loop updates M1, M5, M15, M30, H1, and H4 directly from the same current spot price.
Why this is wrong:
This violates the architecture required in the audits and SRS.
Causally correct higher-timeframe candles must be derived from closed 1m candles, not updated independently in parallel.
Otherwise:
- 5m high/low can disagree with its constituent 1m candles
- 15m close can again inherit next-bucket prices
- data lineage is broken
Required behavior:
- Build only 1m from live spot/tick data.
- Derive higher timeframes from closed 1m candles only.
Required fix:
After each closed 1m candle:
1. persist that 1m candle
2. check whether a 5m boundary completed
3. aggregate exactly the closed 1m rows in that bucket
4. persist derived 5m candle
5. similarly for 15m and above if required
Acceptance gate:
Every 5m candle must exactly match O/H/L/C/V aggregated from its five constituent 1m candles.

ISSUE U-06
Severity: CRITICAL
Title: Backfill forcibly closes partial last candles
Location:
- ETH_G/Engine/CandleEngine.cs lines 75-76
- ETH_G/Db/CandleRepository.cs line 47
Observed behavior:
Backfill inserts candles and then calls CloseAllOpenAsync(), which marks all open rows closed.
In BulkInsertAsync(), is_closed is determined by c.OpenTime < lastTime.
Why this is wrong:
The last bucket in a backfill window may be partial.
Marking it closed makes the system believe the candle is complete when it is not.
That contaminates any downstream strategy logic.
Required behavior:
A bucket should be closed only when completeness is known from the number of constituent lower-timeframe candles or from an explicit data-finalization rule.
Required fix:
- Remove CloseAllOpenAsync() from backfill finalization.
- For each aggregated timeframe, mark a candle closed only if full component coverage exists.
- Add explicit completeness calculation:
  complete if observed_count == expected_count
Acceptance gate:
The final bucket of a range remains open/partial unless it is truly complete.
